# Mem0 MCP Server

<<<<<<< HEAD
> A fully compliant **Model Context Protocol** server that gives any MCP client (AnythingLLM, Claude Desktop, etc.) persistent AI memory backed by **Elasticsearch** as the vector store and **LM Studio** for local LLM + embeddings.

## Architecture

```
MCP Client (AnythingLLM)
        │
        │  POST http://localhost:8004/mcp   (Streamable HTTP MCP)
        ▼
  ┌─────────────┐
  │  mem0 MCP   │  python:3.12-slim  ·  FastMCP + mem0ai
  │  container  │  port 8000 → host 8004
  └──────┬──────┘
         │  Elasticsearch Python client
         ▼
  ┌─────────────────┐        ┌─────────────────────────────┐
  │  Elasticsearch  │        │       LM Studio (host)       │
  │  container      │        │  google/gemma-4-26b-a4b-qat  │
  │  port 9200      │        │  mxbai-embed-large-v1        │
  │  index:         │        │  http://host.docker.internal │
  │  mem0_memories  │        │  :1234/v1                    │
  └─────────────────┘        └─────────────────────────────┘
```

## MCP Tools — Full Reference

### 1. `add_memory`

Extract and persist memories from a conversation for a specific user.

mem0 uses the LLM (Gemma 4 26B) to intelligently extract facts worth remembering from the message list, then stores them as 1024-dimensional embeddings in Elasticsearch. Duplicate or contradictory memories are automatically merged or updated.

**Parameters**

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `messages` | `list[dict]` | ✅ | List of `{"role": ..., "content": ...}` dicts. Roles: `user`, `assistant`, `system` |
| `user_id` | `string` | ✅ | Unique identifier scoping the memories. All searches must use the same ID. |
| `metadata` | `dict` | ❌ | Optional key-value pairs stored with the memory (e.g. `{"source": "chat"}`) |

**Example call**

```json
{
  "messages": [
    {"role": "user", "content": "I always work in dark mode and hate light themes"},
    {"role": "assistant", "content": "Noted, I will remember your UI preferences."}
  ],
  "user_id": "alice",
  "metadata": {"session": "onboarding"}
}
```

**Response**

```json
{
  "results": [
    {
      "id": "3f2a1b4c-8e9d-4c7a-b6f5-1a2b3c4d5e6f",
      "memory": "Prefers dark mode and dislikes light themes",
      "event": "ADD"
    }
  ]
}
```

**Notes**
- `event` is `"ADD"` for new memories, `"UPDATE"` when an existing memory is merged/refined
- mem0 does not store the raw messages — only the extracted facts
- Multiple facts in one conversation may produce multiple memory records

---

### 2. `search_memories`

Semantically search stored memories for a user using vector similarity.

Embeds the query with `mxbai-embed-large-v1`, performs k-nearest-neighbour search in Elasticsearch, and returns the most relevant memories ranked by cosine similarity.

**Parameters**

| Param | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `query` | `string` | ✅ | — | Natural-language question or topic |
| `user_id` | `string` | ✅ | — | Scope to a specific user's memories |
| `limit` | `integer` | ❌ | `10` | Max results returned (recommended: 5–20) |

**Example call**

```json
{
  "query": "What UI preferences does the user have?",
  "user_id": "alice",
  "limit": 5
}
```

**Response**

```json
{
  "results": [
    {
      "id": "3f2a1b4c-...",
      "memory": "Prefers dark mode and dislikes light themes",
      "score": 0.94,
      "metadata": {"session": "onboarding"},
      "user_id": "alice",
      "created_at": "2026-07-12T13:00:00Z",
      "updated_at": "2026-07-12T13:00:00Z"
    }
  ]
}
```

**Notes**
- `score` is cosine similarity (0–1). Results above `0.7` are typically highly relevant.
- Always call this at the **start of a session** to recall user context before responding.

---

### 3. `get_all_memories`

Retrieve every stored memory for a user without filtering.

Use this to present a full memory summary or to audit stored facts. For large memory sets, prefer `search_memories` to stay within LLM context limits.

**Parameters**

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `user_id` | `string` | ✅ | Unique identifier for the user |

**Example call**

```json
{ "user_id": "alice" }
```

**Response**

```json
{
  "memories": [
    {
      "id": "3f2a1b4c-...",
      "memory": "Prefers dark mode and dislikes light themes",
      "metadata": {"session": "onboarding"},
      "user_id": "alice",
      "created_at": "2026-07-12T13:00:00Z",
      "updated_at": "2026-07-12T13:00:00Z"
    },
    { "...": "..." }
  ]
}
```

---

### 4. `get_memory`

Fetch one specific memory record by its UUID.

Use this when you already have the memory ID (from `add_memory` or `search_memories`) and need full detail including metadata and timestamps.

**Parameters**

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `memory_id` | `string` | ✅ | UUID of the memory to retrieve |

**Example call**

```json
{ "memory_id": "3f2a1b4c-8e9d-4c7a-b6f5-1a2b3c4d5e6f" }
```

**Response** — the full memory object, or `{"error": "Memory '...' not found"}` if not found.

---

### 5. `update_memory`

Overwrite the text content of an existing memory.

The new text is re-embedded with `mxbai-embed-large-v1` and the Elasticsearch document is updated in place. The memory's UUID and metadata are preserved; only the text and `updated_at` change.

**Parameters**

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `memory_id` | `string` | ✅ | UUID of the memory to update |
| `data` | `string` | ✅ | New text content to replace the existing memory |

**Example call**

```json
{
  "memory_id": "3f2a1b4c-...",
  "data": "Prefers dark mode and high-contrast themes; also uses Logseq for notes"
}
```

**Response** — the updated memory object, or `{"error": "..."}` on failure.

**Notes**
- Prefer `add_memory` for *new* information — use `update_memory` only to *correct* existing facts.
- The old embedding is discarded and replaced with a fresh one.

---

### 6. `delete_memory`

Permanently delete a single memory by its ID.

Removes both the Elasticsearch document and the embedding vector. The operation cannot be undone.

**Parameters**

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `memory_id` | `string` | ✅ | UUID of the memory to delete |

**Example call**

```json
{ "memory_id": "3f2a1b4c-8e9d-4c7a-b6f5-1a2b3c4d5e6f" }
```

**Response**

```json
{ "deleted": "3f2a1b4c-8e9d-4c7a-b6f5-1a2b3c4d5e6f" }
```

**Notes**
- Call `get_memory_history` first if you want to record what existed before deletion.
- To remove all memories for a user at once, use `delete_all_memories` instead.

---

### 7. `delete_all_memories`

⚠️ **IRREVERSIBLE** — Delete ALL memories belonging to a user.

Removes every Elasticsearch document scoped to this `user_id`, including all embeddings and metadata. There is no undo.

**Typical use cases**
- User requests a "forget me" / GDPR data erasure
- Resetting a test or demo user between runs

**Parameters**

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `user_id` | `string` | ✅ | Unique identifier of the user whose memories to erase |

**Example call**

```json
{ "user_id": "alice" }
```

**Response**

```json
{ "deleted_all_for": "alice", "status": "ok" }
```

---

### 8. `get_memory_history`

Retrieve the full change history of a memory — an audit trail showing every ADD, UPDATE, and DELETE event.

**Use cases**
- Debugging why a memory has unexpected content
- Understanding how the AI's understanding of a user has evolved over time
- Compliance / auditing

**Parameters**

| Param | Type | Required | Description |
|-------|------|----------|-------------|
| `memory_id` | `string` | ✅ | UUID of the memory to inspect |

**Example call**

```json
{ "memory_id": "3f2a1b4c-8e9d-4c7a-b6f5-1a2b3c4d5e6f" }
```

**Response**

```json
{
  "history": [
    {
      "id": "event-uuid-1",
      "memory_id": "3f2a1b4c-...",
      "old_memory": null,
      "new_memory": "Prefers dark mode",
      "event": "ADD",
      "created_at": "2026-07-12T13:00:00Z",
      "updated_at": "2026-07-12T13:00:00Z"
    },
    {
      "id": "event-uuid-2",
      "memory_id": "3f2a1b4c-...",
      "old_memory": "Prefers dark mode",
      "new_memory": "Prefers dark mode and high-contrast themes",
      "event": "UPDATE",
      "created_at": "2026-07-12T14:00:00Z",
      "updated_at": "2026-07-12T14:00:00Z"
    }
  ]
}
```

## Recommended Agent Workflow

```
1. Session start
   └─ search_memories(query="relevant context for this conversation", user_id=user)
   
2. During conversation
   └─ add_memory(messages=[...], user_id=user)   ← after each meaningful exchange

3. Explicit correction
   └─ update_memory(memory_id=id, data="corrected fact")

4. User requests history
   └─ get_memory_history(memory_id=id)

5. User requests erasure
   └─ delete_all_memories(user_id=user)
```

## AnythingLLM System Prompt Snippet

Add this to your workspace system prompt to wire up mem0 automatically:

```
At the start of every conversation, call the mem0 `search_memories` tool with a summary of the topic and user_id="default" to recall relevant context.
After every substantive exchange, call `add_memory` with the last few messages and user_id="default" to preserve what was learned.
Before correcting a stored fact, retrieve it with `get_memory`, then call `update_memory` with the corrected text.
```

## Troubleshooting

| Symptom | Cause | Fix |
|---------|-------|-----|
| `ValidationError: Either api_key or user/password must be provided` | Missing ES creds | Check `ELASTIC_USERNAME`/`ELASTIC_PASSWORD` env vars |
| `ImportError: Elasticsearch requires extra dependencies` | Missing pip package | `mcp[cli]` install should include it; verify compose command |
| `Memory not found` | Wrong `memory_id` or deleted record | Use `get_all_memories` to list valid IDs |
| Embeddings wrong size | Dims mismatch | Confirm `MEM0_EMBEDDER_DIMS=1024`; delete the ES index and restart |
| LLM errors | LM Studio not running | Start LM Studio and load `google/gemma-4-26b-a4b-qat` |
| Embedder errors | Embedder not loaded | Load `text-embedding-mxbai-embed-large-v1` in LM Studio |

## Tools

| Tool | Required arguments | Purpose |
|---|---|---|
| `add_memory` | `messages`, `user_id` | Extract and persist facts |
| `search_memories` | `query`, `user_id` | Semantic retrieval |
| `get_all_memories` | `user_id` | List a user's memories |
| `get_memory` | `memory_id` | Retrieve one memory |
| `update_memory` | `memory_id`, `data` | Correct memory text |
| `delete_memory` | `memory_id` | Delete one memory |
| `delete_all_memories` | `user_id` | Erase a user's memories |
| `get_memory_history` | `memory_id` | Read the audit history |

Use the same `user_id` when adding and searching. The server internally
converts it to the mem0ai API's `filters={"user_id": ...}` parameter.

## LM Studio requirement

Load:

- `google/gemma-4-26b-a4b-qat`
- `text-embedding-mxbai-embed-large-v1`

Set Gemma's context length to **32768**.

## Lifecycle and AnythingLLM compatibility

The MCP app must run with FastMCP's lifespan context. The custom server
mounts `mcp.streamable_http_app()` and passes
`mcp_app.router.lifespan_context` to the outer Starlette application. This
prevents:

```text
Task group is not initialized. Make sure to use run().
```

The server also returns a JSON response for bare GET `/mcp` liveness probes.
Tool calls continue to use POST `/mcp`.

## Recovery

```powershell
docker compose up -d --force-recreate mem0
Invoke-RestMethod http://localhost:8004/health
docker logs mem0 --tail 80
```

After recreation, reconnect mem0 in AnythingLLM to establish a fresh session.
